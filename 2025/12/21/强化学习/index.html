PPO

近端策略优化，其实就是利用奖励模型的结果更新策略模型参数，既希望语言模型的输出能够尽可能地获得奖励模型的高评价，又不希望语言模型的变化过于“激进”

策略 →（生成）→ 响应 →（输入）→ 奖励模型 →（输出）→ 奖励 →（反馈）→ 策略

组件

策略模型（Actor）

通常是大型语言模型（如 LLaMA、GPT 等），也就是经过预训练或SFT的LLM，负责生成内容，后续是可以更新参数的

参考模型（Reference）

经过预训练或SFT的模型，与初始的策略模型一致，但是参数冻结，无法更新，目的是为了约束策略模型的更新

奖励模型（Reward）

奖励模型是由人类标注的偏好数据集训练得到的模型，作用是对模型输出进行排序，预测人类偏好，用来代替人类对语言模型的输出进行评价。

提供的是智能体在某一状态下执行某个动作后的即时反馈信号

价值函数（Critic）

用于评估智能体处于某个状态或执行某个 动作-状态对 时的“好坏”，即能够获得的长期回报（累积奖励）

特性 奖励模型 价值函数
是什么 一个函数 R(s, a, s')，表示在状态 s 采取动作 a 到达状态 s’ 后，立即获得的反馈信号。 一个函数 V(s) 或 Q(s, a)，表示从当前状态（或状态-动作对）出发，所能获得的长期、累积回报的期望。
时间视角 短视的、瞬时的。只关心“这一步”的直接结果。 远见的、长期的。关心“从今往后”的所有可能结果。
核心问题 “这个动作现在有多好？” “处于这个状态/采取这个动作，未来能有多好？”
人类类比 感官愉悦/痛苦：吃糖的甜味（+奖励），摸火炉的灼痛（-奖励）。 战略判断：选择读大学（现在辛苦，但长期价值高），选择打游戏（现在快乐，但可能损害长期价值）。
在RL中的角色 环境的反馈机制，定义了任务的终极目标。是价值函数计算的原始输入。 智能体的内部认知模型，是进行决策和规划的核心依据。

广义优势估计 (GAE)

它定义了在状态 st（提示 + 到目前为止生成的词）下，特定动作 at（词）相对于策略将采取的平均动作的优越程度

两种计算优势的方法：

  • 蒙特卡洛 (MC)：使用完整轨迹的奖励，高方差低偏差
  • 时序差分 (TD)：每一步都会得到一个即时奖励，低方差高偏差

GAE 的目标就是：在这两者之间取得一个可调节的平衡。通过指数加权平均多个 n-step TD（多步TD） 误差，构造一个偏差与方差可控的优势估计量。

计算方法

这一步的优势 = 这一步的即时奖励 + 新预测的最终成绩 - 旧预测的最终成绩

  • R_t：在状态 s_t 采取行动后立即获得的奖励
  • V(s_t)：在状态 s_t 预测从此刻到结束能获得的总奖励
  • γ：折现因子（0≤γ≤1），通常接近1
  • V(s_{t+1})：进入新状态后的总奖励预测
λ 取值 含义 等价于
λ = 0 只看当前一步 TD error TD(0)
0 < λ < 1 TD 与 MC 的折中 GAE
λ = 1 看完整轨迹 MC Advantage

A > 0： 👉 这次回答 比 critic 预期的好 👉 应该鼓励 actor 更常这么写

A < 0： 👉 这次回答 比预期的差 👉 应该抑制 actor 再这么写

A ≈ 0： 👉 和预期差不多 👉 别怎么动参数

示例

场景：AI生成句子“我喜欢吃苹果。”

  • 第1步：生成“我”
    • 价值函数（预言家）预测最终奖励：预测分数 = 0.5
  • 第2步：生成“喜欢”
    • 即时奖励（TD）：+0.3（因为“我喜欢”是常见搭配）
    • 价值函数重新预测：预测分数 = 0.8（因为开头不错，提高了最终预期）
    • GAE计算的这一步的优势：0.3（即时奖励）+ 0.8（新预测）- 0.5（旧预测）= +0.6
    • 解读：生成“喜欢”这个词很棒！不仅本身得分高（+0.3），还让预言家对最终结果的预测提升了0.3分，总优势为+0.6。
  • 第3步：生成“吃”
    • 即时奖励：+0.2
    • 价值函数重新预测：预测分数 = 0.9
    • 优势：0.2 + 0.9 - 0.8 = +0.3
  • 第4步：生成“苹果”
    • 即时奖励：+0.1
    • 价值函数重新预测：预测分数 = 1.0（句子完整且合理）
    • 优势：0.1 + 1.0 - 0.9 = +0.2

裁剪

防止策略修正过头，例如“限定这次更新里，每个 token 的概率最多只能改 ±20%”

概率比的计算中，只有response参与 \[ r_t(\theta) = \frac{\pi_\theta(y_t \mid s_t)} {\pi_{\theta_{\text{old}}}(y_t \mid s_t)} = \exp(\log p_t^{\text{new}} - \log p_t^{\text{old}}) \]

KL散度

防止当前策略 θ 与我们在微调时所基于的原始模型 θorig 偏离太远

熵奖励

鼓励LLM生成结果具有多样性

总体流程

  1. 用当前 policy 生成完整 response（rollout）

  2. 用 Reward Model 给整段 response 打分（一个标量）

  3. 用 value head 给每个 token 计算 V(s_t)

    “PPO 只看我选了的这个 token，那没选的那些 token 怎么知道是好是坏？”

    结论：PPO / policy gradient 根本“不试图知道”没选的 token 好不好。 它只做一件事： 👉 如果你选了这个 token，结果好，就提高它； 👉 结果不好，就压低它。

  4. 用 GAE 计算每个 token 的 advantage A_t

  5. 用所有 token 的 (logprob, A_t) 计算 PPO loss

  6. 对 policy / value 网络做反向传播（梯度下降）

on-policy RL

RL在工程上的流程，这里以PPO为例

阶段 1：Rollout / Generation（关键路径）

这是 on-policy RLHF 的起点，不能并行跳过

模型：Actor（Generator）

输入

  • prompt

输出（在生成时同步得到）

  • response
  • old_log_prob（token 级）：旧策略对该 token 的 log 概率，在采样时计算

阶段 2：并行评估（模型 forward 层）

response 已存在，下面这些模型之间无数据依赖

2.1 Critic（Value Model）

输入

  • prompt + response

输出

  • value（通常 token 级）

2.2 Reference Model（参考模型，参数冻结）

输入

  • prompt + response

输出

  • ref_log_prob（token 级）：用于防止actor偏离太多

2.3 Reward Model

输入

  • prompt + response

输出

  • rm_score（标量或 token 级）

📌 这三者：

  • 可以并行
  • 是系统优化重点
  • 不影响 on-policy 正确性
