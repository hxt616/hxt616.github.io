Python 多进程：fork、spawn 与 main

发表于 2026-06-09 分类于 Python 阅读次数：
本文字数： 2.8k 阅读时长 ≈ 3 分钟

在 Python 中进行多进程（multiprocessing）开发时，很多人经常会被跨平台报错、死锁或者莫名的无限递归卡住。要彻底解决这些问题，我们需要理清三个核心概念：进程启动方式、上下文（Context）以及为什么必须加 if __name__ == '__main__':。

一、 进程创建流程（Fork vs Spawn）

Python 启动一个新进程主要有两种底层机制，它们的区别决定了代码的安全性和执行效率。

1. Fork

  • 怎么工作：这是 Linux/Unix 系统的默认方式。它就像是“克隆”，当主进程要开新进程时，操作系统把主进程现有的内存、变量、打开的文件等原封不动地复制一份给子进程。
  • 优缺点：因为不需要重新加载代码，它的启动速度极快。但是，它有一个致命隐患——死锁。如果主进程里有很多线程，fork 时某个后台线程刚好给某个资源上了锁，子进程会把这个“锁死”的状态一起复制过去。但负责开锁的线程在子进程里不存在，子进程一旦访问这个资源，就会永久卡死。

2. Spawn

  • 怎么工作：这是 Windows 和 macOS 的默认方式。它不搞克隆，而是白手起家。主进程会启动一个完全干净、全新的 Python 解释器。子进程为了知道自己要干什么，必须把你的 Python 脚本重新读取（import）一遍才能获取要执行的函数，并且只能通过序列化（Pickle）接收主进程传过来的核心数据。
  • 优缺点：因为要重新加载环境和代码，启动速度比 fork 慢。但它极其安全，进程之间完全隔离，彻底避免了死锁问题。

行业趋势：由于 fork 的安全性隐患，现代 Python（包括 macOS 和未来的 Linux 版本）都在默认或者全面转向更安全的 spawn。

产生错误的过程：

  1. 主进程启动：直接运行脚本，此时 __name__ 的值是 __main__。代码向下跑，执行到 process.start()。主进程通过 spawn 方式告诉系统：“帮我开一个全新的子进程。”
  2. 子进程诞生：子进程是一个完全空白的 Python 解释器。它在脑子里找不到你要运行的那个函数。为了把函数加载到内存里，子进程必须把你的这个脚本从头到尾重新导入（import）一遍。
  3. 连环爆炸：子进程在重新读取脚本时，因为没有 if __name__ == '__main__': 的拦截，它会老老实实地把脚本里的代码再执行一遍。当它执行到 process.start() 时，子进程也会向系统申请：“帮我再开一个全新的孙子进程。”
  4. 无限循环：孙子进程启动，为了加载函数，又重新导入脚本，又创建重孙进程……在几毫秒内，系统会陷入无限递归创建进程的死循环，直到系统崩溃。

防止错误

当你加上 if __name__ == '__main__': 之后：

  1. 主进程满足条件，成功创建子进程。
  2. 子进程为了加载函数，重新导入（import）脚本。
  3. 导入时，子进程内部的 __name__ 变量变成了当前的文件名，不再是 __main__。
  4. 于是子进程卡在 if 判断外面，创建新进程的代码被完美拦截。子进程顺利加载完函数，高高兴兴去干活了。
