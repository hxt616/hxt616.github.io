问题分析

写入操作是非合并的

结果：32个线程的写入地址分散在内存的各个角落，每个访问都需要独立的全局内存事务 → 带宽利用率可能低至 5%。

二、共享内存版本

关键：

  • 读取：从输入矩阵是按行访问的（合并的，快）
  • 写入：到输出矩阵是按列访问的（非合并的，慢）

共享内存的作用：作为中转站，解耦读写访问模式 1. 先按行读入共享内存（合并读） 2. 在共享内存中完成转置（内部重排，不涉及全局内存） 3. 再从共享内存按行写出（合并写）

五、性能对比分析

全局内存事务数量对比（width=1024, tile=32）

版本 读取事务 写入事务 总事务 带宽效率
朴素版本 1024×1024/32 = 32,768 32,768（非合并） 65,536 ~10-20%
共享内存版本 32,768（合并） 32,768（合并） 65,536 ~90-100%

虽然总事务数相同，但： - 朴素版本：每次写入事务传输 32 字节，但只用了 4 字节（1个float）→ 浪费 87.5% 带宽 - 优化版本：每次写入事务传输 32 字节，32 字节全部有效 → 充分利用带宽

六、为什么不直接按列读、按行写？

你可能会想：那我可以改变读取模式来匹配写入吗？

问题：按列读取同样是非合并的，只是把瓶颈从写入转移到了读取。总性能不会改善。

共享内存的优雅之处在于：让读取和写入都变成合并的，通过中间存储解耦了输入和输出的访问模式。
